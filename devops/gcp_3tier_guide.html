  • 필수 API 활성화
    상단 검색창 → "Compute Engine API" 검색 → 활성화
  • gcloud CLI 설치 (선택사항)
    로컬에서 GCP 관리하려면
    $ curl https://sdk.cloud.google.com | bash $ gcloud init $ gcloud auth login

1
1-tier: Nginx 배포 (Frontend)
🌐 역할: 정적 파일 제공 + 로드밸런싱 사용자의 HTTP 요청을 받아 Vue.js 파일을 제공하고, /api/* 요청을 Spring Boot로 전달합니다. 여러 Boot 인스턴스에 요청을 분산(로드밸런싱)합니다.

Step 1-1: VPC 네트워크 생성

  • VPC 생성
    좌측 메뉴 → VPC 네트워크 → VPC 네트워크 생성
  • 설정값
    항목 값
    네트워크 이름 3tier-network
    서브넷 모드 자동
    생성 클릭

Step 1-2: 방화벽 규칙 설정

  • HTTP 트래픽 허용
    VPC 네트워크 → 방화벽 → 규칙 만들기
    항목 값
    이름 allow-http
    네트워크 3tier-network
    우선순위 1000
    방향 수신
    소스 IP 범위 0.0.0.0/0
    프로토콜 및 포트 TCP 80, 443
  • 내부 통신 허용
    같은 과정으로 또 다른 규칙 생성
    항목 값
    이름 allow-internal-8080
    소스 IP 범위 10.0.0.0/8
    프로토콜 및 포트 TCP 8080

Step 1-3: Nginx VM 생성

  • 인스턴스 생성
    좌측 메뉴 → Compute Engine → 인스턴스 → 만들기
  • 설정값
    항목 값
    이름 nginx-vm
    리전 us-central1 (항상 무료)
    영역 us-central1-a
    머신 타입 e2-micro (항상 무료)
    이미지 Ubuntu 22.04 LTS
    부팅 디스크 30GB
    네트워크 3tier-network
  • 생성 대기
    VM이 시작될 때까지 대기 (1-2분)
✓ 확인
nginx-vm 우측의 SSH 버튼이 활성화되었는지 확인

Step 1-4: Nginx 설치 & 설정

  • VM에 SSH 접속
    nginx-vm 우측 → SSH 버튼 클릭
  • Nginx 설치
    터미널에서 실행
    $ sudo apt-get update $ sudo apt-get install -y nginx $ sudo systemctl start nginx $ sudo systemctl enable nginx
  • Nginx 설정 파일 편집
    편집 시작
    $ sudo nano /etc/nginx/sites-available/default
    아래 내용으로 전체 교체:
    upstream spring_boot { server 10.0.0.20:8080; # Boot VM-1 server 10.0.0.21:8080; # Boot VM-2 } server { listen 80; server_name _; # Vue.js 정적 파일 location / { root /var/www/html; try_files $uri $uri/ /index.html; } # Spring Boot API location /api/ { proxy_pass http://spring_boot; proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; } }
  • Ctrl+X → Y → Enter로 저장
    nano 에디터 종료
  • Nginx 재시작
    $ sudo nginx -t $ sudo systemctl restart nginx

Step 1-5: Vue.js 배포

  • 로컬에서 Vue.js 빌드
    로컬 터미널
    $ cd your-vue-project $ npm run build # dist/ 폴더 생성됨
  • VM으로 파일 전송
    로컬에서
    $ gcloud compute scp --recurse dist/* \ nginx-vm:/tmp/ --zone=us-central1-a
  • VM에서 배포
    nginx-vm SSH 터미널
    $ sudo cp -r /tmp/* /var/www/html/ $ sudo chown -R www-data:www-data /var/www/html/ $ sudo systemctl reload nginx
✓ 완료
nginx-vm의 외부 IP로 브라우저 접속 → Vue.js 앱이 로드되는지 확인
2
2-tier: Spring Boot 배포 (Backend)
⚙️ 역할: API 제공 + 비즈니스 로직 Nginx에서 받은 /api/* 요청을 처리하고, H2 메모리 DB에 데이터를 저장/조회합니다. 2개 VM에서 독립적으로 실행됩니다.

Step 2-1: Spring Boot 프로젝트 준비

  • pom.xml 의존성 추가
    <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-web</artifactId> </dependency> <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-data-jpa</artifactId> </dependency> <dependency> <groupId>com.h2database</groupId> <artifactId>h2</artifactId> <scope>runtime</scope> </dependency>
  • application.properties 설정
    src/main/resources/application.properties
    # H2 메모리 DB spring.datasource.url=jdbc:h2:mem:testdb spring.datasource.driverClassName=org.h2.Driver spring.datasource.username=sa spring.datasource.password= # JPA spring.jpa.database-platform=org.hibernate.dialect.H2Dialect spring.jpa.hibernate.ddl-auto=create-drop # Server server.port=8080
  • Entity 생성
    User.java
    @Entity public class User { @Id @GeneratedValue private Long id; private String name; private String email; // getter/setter }
  • Repository 생성
    UserRepository.java
    @Repository public interface UserRepository extends JpaRepository<User, Long> { }
  • Controller 생성
    UserController.java
    @RestController @RequestMapping("/api/users") public class UserController { @Autowired private UserRepository userRepo; @GetMapping public List<User> getAll() { return userRepo.findAll(); } @PostMapping public User create(@RequestBody User user) { return userRepo.save(user); } }

Step 2-2: Spring Boot 빌드 & 배포

  • 로컬에서 빌드
    로컬 터미널
    $ mvn clean package # target/app.jar 생성 (또는 실제 빌드 결과 jar 파일)
  • Boot VM-1, VM-2 생성
    Nginx VM과 동일한 방식으로 2개 VM 생성
    설정 값
    이름 boot-vm-1, boot-vm-2
    머신 타입 e2-micro
    이미지 Ubuntu 22.04 LTS
    네트워크 3tier-network
    외부 IP 없음 (내부 통신만)
  • JAR 파일 전송
    로컬에서
    $ gcloud compute scp target/app.jar \ boot-vm-1:/tmp/ --zone=us-central1-a $ gcloud compute scp target/app.jar \ boot-vm-2:/tmp/ --zone=us-central1-a

Step 2-3: Boot VM에 Java 설치 & 실행

  • boot-vm-1 SSH 접속
    GCP 콘솔 → boot-vm-1 → SSH
  • Java 설치
    $ sudo apt-get update $ sudo apt-get install -y openjdk-21-jre
  • Systemd 서비스 등록
    /etc/systemd/system/springboot.service 생성
    $ sudo nano /etc/systemd/system/springboot.service
    아래 내용 입력:
    [Unit] Description=Spring Boot Application After=network.target [Service] Type=simple User=ubuntu WorkingDirectory=/home/ubuntu/app ExecStart=/usr/bin/java -jar app.jar Restart=always [Install] WantedBy=multi-user.target
  • 서비스 시작
    $ sudo cp /tmp/app.jar /home/ubuntu/app/ $ sudo chown ubuntu:ubuntu /home/ubuntu/app/app.jar $ sudo systemctl daemon-reload $ sudo systemctl enable springboot $ sudo systemctl start springboot $ sudo systemctl status springboot # 확인
  • boot-vm-2도 동일하게 설정
    boot-vm-2에 SSH 접속해서 같은 절차 반복
✓ 완료
두 VM 모두 systemctl status springboot에서 "active (running)" 표시되는지 확인
3
통합 테스트 & 검증
🧪 목표: 전체 아키텍처가 제대로 동작하는지 확인 Nginx를 통한 요청, 로드밸런싱, 데이터 격리 등을 검증합니다.

Step 3-1: 기본 연결 확인

  • Vue.js 앱 로드 확인
    브라우저에서 nginx-vm 외부 IP 접속 → Vue.js 화면이 로드되는지 확인
  • API 응답 확인
    터미널에서 (또는 브라우저)
    $ curl http://NGINX_EXTERNAL_IP/api/users → [] # 빈 배열 (아직 데이터 없음)

Step 3-2: 로드밸런싱 확인

  • 각 Boot VM에 서버 식별자 추가
    다시 각 Boot VM의 application.properties에 추가:
    # application.properties 하단에 추가 SERVER_ID=boot-vm-1 (boot-vm-1에는 boot-vm-1, boot-vm-2에는 boot-vm-2)
    또는 controller에 추가:
    @RestController @RequestMapping("/api") public class InfoController { @Value("${SERVER_ID:unknown}") private String serverId; @GetMapping("/server") public Map<String,String> getServer() { return Map.of("server", serverId); } }
  • 여러 번 요청하면 VM이 번갈아 응답하는지 확인
    여러 번 실행
    $ curl http://NGINX_IP/api/server → {"server":"boot-vm-1"} $ curl http://NGINX_IP/api/server → {"server":"boot-vm-2"} $ curl http://NGINX_IP/api/server → {"server":"boot-vm-1"}

Step 3-3: 데이터 격리 확인

  • boot-vm-1에만 데이터 생성
    boot-vm-1의 내부 IP로 직접 요청
    $ curl -X POST http://10.0.0.20:8080/api/users \ -H "Content-Type: application/json" \ -d '{"name":"Alice","email":"alice@test.com"}' → {"id":1,"name":"Alice",...}
  • boot-vm-1에서 조회 (데이터 있음)
    $ curl http://10.0.0.20:8080/api/users → [{"id":1,"name":"Alice",...}]
  • boot-vm-2에서 조회 (데이터 없음!)
    $ curl http://10.0.0.21:8080/api/users → [] # 빈 배열 (H2 메모리는 독립적)
    이것이 의도된 동작입니다!
⚠️ 중요
boot-vm-1과 boot-vm-2는 각각 독립적인 H2 메모리 DB를 사용합니다. 데이터가 공유되지 않는 것이 정상입니다. 이는 학습 목적이기 때문에 의도된 동작입니다.

Step 3-4: Nginx를 통한 로드밸런싱 테스트

  • Nginx를 통해 여러 요청 후 조회
    Nginx IP를 통한 요청
    $ curl -X POST http://NGINX_IP/api/users \ -H "Content-Type: application/json" \ -d '{"name":"Bob","email":"bob@test.com"}' # 응답이 boot-vm-1 또는 boot-vm-2에서 올 수 있음
  • 이전에 생성한 Alice 데이터 확인
    $ curl http://NGINX_IP/api/users → [{"id":1,"name":"Alice",...}] 또는 []
    라운드로빈 로드밸런싱으로 VM-1 또는 VM-2로 가므로 응답이 다를 수 있습니다.
✓ 성공!
• Nginx가 정적 파일(Vue.js)을 제공 ✓
• 로드밸런싱이 동작 (요청이 VM-1/VM-2에 번갈아 감) ✓
• 각 VM이 독립적 H2 메모리 DB 사용 ✓
