GCP 3-tier 아키텍처 개념 설명

각 단계가 왜 필요한지, 어떻게 동작하는지 이해하기

1. 전체 아키텍처 구조

2. 1-tier: Nginx (Frontend 계층)

역할

🌐 정적 콘텐츠 서빙 + 동적 요청 라우팅 사용자의 HTTP 요청을 받아서, 정적 파일(Vue.js)은 직접 제공하고, API 요청은 Spring Boot로 전달합니다.

왜 필요한가?

✓ 성능 분리
정적 파일 서빙은 가볍고 빠르기 때문에 Nginx 같은 웹 서버가 최적화. Spring Boot는 비즈니스 로직에 집중.
✓ 로드밸런싱
여러 Spring Boot 인스턴스 중 하나를 자동으로 선택. 한 서버가 느려도 다른 서버로 분산.
✓ 단일 진입점 (Entry Point)
사용자는 Nginx IP 하나만 알면 됨. 뒤에 여러 Spring Boot 서버가 있어도 투명하게 동작.
✓ 보안
Spring Boot를 외부에 노출하지 않음. Nginx가 방화벽 역할.

데이터 흐름

사용자 브라우저 ↓ GET / (Vue.js HTML) Nginx ↓ /var/www/html/index.html 읽음 ↓ HTML 응답 사용자 브라우저 ↓ GET /api/users (JavaScript 요청) Nginx ↓ /api/* → upstream spring_boot ↓ Spring Boot VM-1 또는 VM-2로 전달 ↓ JSON 응답

3. 2-tier: Spring Boot (Application 계층)

역할

⚙️ 비즈니스 로직 + 데이터 처리 사용자 요청을 받아서 데이터를 조회/생성/수정하고, JSON으로 응답합니다.

멀티 인스턴스 구성 (VM-1, VM-2)

📊 로드밸런싱 테스트 목적
같은 애플리케이션을 여러 VM에서 실행하여, 부하 분산과 확장성을 학습합니다.

각 인스턴스의 특성

항목 Boot VM-1 Boot VM-2
내부 IP 10.x.x.y 10.x.x.z
포트 8080 8080
H2 DB 독립 메모리 독립 메모리
데이터 공유 ✗ 없음 ✗ 없음

데이터 흐름

Nginx (라운드로빈) ↓ 요청 1번째 Boot VM-1 ↓ H2 메모리 DB (VM-1 독립) ↓ 데이터 저장/조회 Nginx (라운드로빈) ↓ 요청 2번째 Boot VM-2 ↓ H2 메모리 DB (VM-2 독립) ↓ 다른 데이터 상태

⚠️ 주의: 데이터 격리

각 Spring Boot 인스턴스가 독립적 메모리 DB 사용

VM-1: User 테이블에 Alice, Bob
VM-2: User 테이블에는 아무 데이터 없음

→ 이는 "학습 목적"이기 때문에 의도된 동작
→ 실제 서비스는 데이터 공유 필수 (Firestore 전환 필요)

4. 3-tier: H2 메모리 DB (Data 계층)

위치 & 저장소

💾 Spring Boot 애플리케이션 내부 메모리 별도의 DB 서버가 아니라, 각 Spring Boot 프로세스의 RAM에 직접 저장됩니다.

H2 메모리 DB vs 다른 방식

방식 저장 위치 재시작 후 멀티 인스턴스 비용
H2 메모리 RAM 데이터 소실 공유 불가 $0
H2 파일 디스크 유지 같은 VM만 $0
Cloud SQL 관리형 DB 유지 모두 공유 $6.50+
Firestore 관리형 DB 유지 모두 공유 $0

메모리 특성 이해하기

🧠 RAM = 매우 빠르지만 휘발성

• 속도: 디스크보다 수천 배 빠름
• 용량: 보통 1-2GB 정도만 사용 가능
• 휘발성: 전원 끄면 사라짐 (앱 재시작도 포함)

→ 학습/테스트에는 최적
→ 실제 서비스는 불가능

생명 사이클

Spring Boot 시작 ↓ H2 메모리 DB 초기화 ↓ (spring.jpa.hibernate.ddl-auto=create-drop) 테이블 자동 생성 ↓ 데이터 저장/조회/수정/삭제 ↓ Spring Boot 중지 ↓ 모든 데이터 사라짐 (RAM이므로) ↓ 다시 시작 ↓ 빈 DB부터 시작

5. VPC 네트워크: VM 간 통신

왜 VPC가 필요한가?

✓ 프라이빗 네트워크
Nginx ↔ Spring Boot 간 통신이 내부 네트워크(10.x.x.x)를 통해 이루어지므로, 외부에 노출되지 않음.
✓ 빠른 통신
같은 VPC 내 VM들 간의 통신은 지연이 적고 비용도 없음.
✓ 보안 그룹 (방화벽)
어떤 포트만 허용할지 세밀하게 제어 가능.

6. 전체 데이터 흐름

시나리오: 사용자가 새로운 사용자를 추가하는 경우

1. 사용자 브라우저 POST /api/users (JSON: {name:"Alice"}) 2. Nginx (외부 IP) ↓ 요청 받음 3. Nginx 리버스 프록시 upstream spring_boot 확인 → Boot VM-1 선택 (라운드로빈) 4. Boot VM-1 (내부 IP 10.0.0.20:8080) UserController.create() 실행 ↓ 5. Boot VM-1의 H2 메모리 DB INSERT INTO users(name) VALUES('Alice') ↓ 6. Boot VM-1 응답 {id:1, name:"Alice", ...} 7. Nginx 응답 Nginx → 브라우저로 전달 8. 사용자 화면 새로운 사용자 표시

다음 요청이 Boot VM-2로 간다면?

⚠️ Boot VM-2의 H2는 독립적
GET /api/users → Boot VM-2로 라우팅
→ VM-2의 H2 DB 조회
→ 아무 데이터도 없음 (Alice는 VM-1에만 있음)
→ 빈 배열 [] 응답

7. Nginx 로드밸런싱: 왜 필요한가?

로드밸런싱 없을 때

❌ 한 서버에만 모든 부하

Boot VM-1: CPU 100%, 느림
Boot VM-2: 놀고 있음

→ 전체 시스템이 느려짐

로드밸런싱 있을 때

✓ 부하 분산

요청 1번, 3번, 5번 → Boot VM-1
요청 2번, 4번, 6번 → Boot VM-2

→ 두 서버가 균등하게 일함
→ 전체 응답 시간 단축

Nginx 라운드로빈 방식

upstream spring_boot { server 10.x.x.y:8080; # Boot VM-1 server 10.x.x.z:8080; # Boot VM-2 } # 요청이 순서대로 분배됨 요청 1번 → VM-1 요청 2번 → VM-2 요청 3번 → VM-1 요청 4번 → VM-2 ...

8. 학습 구성 vs 실제 서비스

항목 지금 당신의 구성 (학습) 실제 서비스
Database H2 메모리 (독립) Firestore 또는 Cloud SQL (공유)
데이터 지속성 ✗ 재시작 시 소실 ✓ 영구 저장
멀티 인스턴스 ✗ DB 공유 불가 ✓ 데이터 일관성
비용 $0 $6.50+ (Cloud SQL) 또는 $0 (Firestore)
목적 개념 학습 실제 운영
