Transaction 완벽 가이드

데이터베이스 트랜잭션의 개념과 동작 원리 정리

1. Transaction 정의

Transaction이란?
개별 쿼리 단위가 아닌 사용자의 액션(업무) 단위로 묶어서 제어하는 데이터 처리의 한 단위

목적

데이터베이스 내 데이터의 완전성(integrity) 유지

실제 예시: 계좌이체

A 계좌 100원 → B 계좌로 이체

UPDATE 1: A 계좌에서 100원 출금

UPDATE 2: B 계좌에 100원 입금

→ 이 2개의 SQL을 하나로 묶은 것이 "이체"라는 업무

⚠️ A 계좌 출금 성공 후 B 계좌 입금 실패 시?
→ 출금을 반드시 취소(Rollback)해야 함

2. Commit과 Rollback

동작 원리

핵심 원칙: 트랜잭션 종료 전 모든 변경사항은 임시 데이터

변경 전 데이터는 UNDO Tablespace에 보관되어 언제든 복구 가능

처리 과정

UPDATE 실행
→
UNDO에 기존값 저장
→
테이블에 임시값 저장

Commit

  • 임시 데이터를 확정
  • UNDO의 값 삭제 가능 (정책에 따라 다름)
  • 업무 완전 성공

Rollback

  • UNDO의 데이터로 테이블 덮어쓰기
  • 변경사항 모두 취소
  • 업무 실패로 원래 상태 복원
All or Nothing 원칙: 트랜잭션은 완전히 성공하거나 완전히 실패해야 함

3. Transaction의 단계

1트랜잭션 시작

START TRANSACTION;

또는 DML 실행 시 자동 시작

2종료 전 데이터 상태

  • 모든 데이터: 임시적 상태
  • 변경 전 데이터로 언제든 복구 가능
  • 변경 TX: 해당 row에 lock 확보 → 조회/수정/삭제 자유
  • 다른 TX: 편집 불가 (읽기는 상황에 따라 다름)

3트랜잭션 종료

  • COMMIT: 변경사항 확정
  • ROLLBACK: 변경사항 취소
  • 자동 COMMIT: DDL(create, alter, drop) 또는 DCL(grant, revoke) 실행 시
  • 자동 ROLLBACK: SQL 도구 비정상 종료 시

4. A.C.I.D 원칙

트랜잭션이 안전하게 처리되기 위해 보장되어야 하는 4가지 특성

A
Atomicity (원자성)
작업이 부분적으로 성공하면 안됨
→ 하나로 움직여야 함
→ All or Nothing

예: A 계좌 출금 성공
    B 계좌 입금 실패
    → 출금도 취소 (ROLLBACK)
C
Consistency (일관성)
데이터에 일관성 유지
→ 이체 전후 금액 합 변화 없음
→ 자료형 임의 변경 금지

예: 이체 전: A 100원, B 200원
    이체 후: A 0원, B 300원
    → 총합 300원 유지
I
Isolation (독립성)
다른 TX에 간섭 없음
→ TX 중간 내용 침범 금지
→ 다른 이체 개입 금지

예: TX A 이체 중
    TX B가 A 잔액 조회
    → 이체 완료 후 결과만 보임
D
Durability (지속성)
완료된 TX는 영구적
→ 시스템 장애 후에도 복구
→ 송금 결과 반영 유지

예: 이체 COMMIT 후
    시스템 장애 발생
    → 재부팅 시에도 이체 결과 유지

5. Lock의 개념

필요성

A.C.I.D 원칙 완벽 보장 방법: 한번에 하나의 TX만 동작

⚠️ 문제: 동시성 저하 → 성능 심각 악화

해결책: Lock을 통한 효율적 자원 제어

Lock의 역할

동시에 진행되는 여러 TX 중 특정 자원에 대한 접근 제어

  • 대상 수정 TX: 그 권한 확보
  • 다른 TX: 사용에 제약
  • 순차성 보장 → 무결성 유지

6. Lock의 분류

① 설정 범위별 분류

테이블 Lock 테이블 전체에 걸리는 Lock → DDL 과정에서 발생 (ALTER 등)
Row Lock 하나의 행에 걸리는 Lock → DML의 Lock → 가장 일반적

② 동작 방식별 분류

🔓 Shared Lock (공유 락, S Lock)

사용 시점: SELECT 조회 과정에서 부여

특징: 여러 사용자가 동시에 같은 데이터 읽기 가능

효과: Dirty Read, Non-Repeatable Read 방지

🔒 Exclusive Lock (배타 락, X Lock)

사용 시점: INSERT, UPDATE, DELETE 등 변경 작업 시

특징: 해당 TX 완료까지 다른 세션 완전 차단

효과: 변경 중 충돌 방지 → 데이터 무결성 유지

7. Lock 동작 시나리오

① Shared Lock (S-Lock) 시나리오

여러 TX가 동시에 같은 데이터를 읽기만 할 때의 동작

⚠️ 주의: 아래 시나리오는 SERIALIZABLE Isolation Level에서의 동작입니다.
일반적인 DB 기본값(READ COMMITTED)에서는 S-Lock이 SELECT 직후 즉시 해제되어 TX B의 UPDATE가 바로 성공합니다.
Transaction A Transaction B 설명 (SERIALIZABLE)
START TRANSACTION;
SELECT * FROM account;		 — TX A: Shared Lock (S Lock) 획득 → COMMIT까지 유지
— START TRANSACTION;
SELECT * FROM account;		 TX B: S Lock 획득 → 성공 (S Lock끼리 충돌 없음)
— UPDATE account SET amount=200
WHERE no=1;		 TX B: X Lock 시도 → 대기 (TX A의 S Lock 때문에)
COMMIT; — TX A: S Lock 반납 → TX B UPDATE 진행 가능
— UPDATE 성공
COMMIT;		 TX B: X Lock 획득 후 수정 완료
S-Lock의 개념 여러 TX 동시 읽기 가능 ↑ 동시성 / Isolation Level에 따라 Lock 유지 기간 달라짐 → 실제 동작 상이

② Exclusive Lock (X-Lock) 시나리오

특정 행을 변경할 때의 동작 (다른 TX 완전 차단)

Transaction A Transaction B 설명
START TRANSACTION;
SELECT * FROM account;		 — TX A: 초기 상태
UPDATE account SET amount=200
WHERE no=1;		 — TX A: Exclusive Lock (X Lock) 획득 → no=1 row 잠금
— START TRANSACTION;
SELECT * FROM account;		 TX B: 조회는 가능 (상황에 따라 다름)
— UPDATE account SET amount=200
WHERE no=1;		 TX B: X Lock 획득 시도 → 대기 (Lock timeout)
COMMIT; — TX A: Lock 반납 → TX B 진행 가능
— UPDATE 성공
SELECT * FROM account;		 TX B: X Lock 획득 후 수정 완료
X-Lock 특성 다른 TX 완전 차단 ↓ 동시성 낮음 / 데이터 무결성 최우선 ↑ 안정성 높음

8. 핵심 포인트

독립성 문제 A.C.I.D 중 독립성 보장이 실제로 가장 어려움
→ 무결성과 동시성의 균형 필요
→ 이를 위해 Isolation Level 개념 등장
성능과 안정성 너무 강한 Lock: 안정적이지만 성능 악화
→ 동시성 저하, 대기 시간 증가
→ 상황에 맞는 적절한 Lock 전략 필요
