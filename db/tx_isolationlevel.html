Transaction Isolation Level

동시성과 안정성의 트레이드오프 - 4가지 레벨 완벽 분석

1. Isolation Level 개요

문제 상황: Lock이 너무 강하면 성능 악화

→ 해결책: 동시에 여러 TX가 처리될 때, 다른 TX의 변경 중/조회 중인 데이터를 읽을 것인가?를 결정

목표: 독립성(안정성)과 동시성의 적절한 균형

4가지 Isolation Level

← 왼쪽이 약함(동시성 높음) | 오른쪽이 강함(안정성 높음) →

1️⃣ READ UNCOMMITTED (가장 약함)
COMMIT/ROLLBACK 여부 무관 → 모든 데이터 읽기 가능
✅ 동시성 최고 | ❌ 안정성 최악
Oracle 비추천 | MySQL 비추천
2️⃣ READ COMMITTED
COMMIT된 데이터만 읽기 가능
✅ 일반적 수준 | ✅ Oracle 기본값
문제점: Non-Repeatable Read, Phantom Read
3️⃣ REPEATABLE READ
한번 읽은 데이터는 TX 종료까지 변함없음
✅ MySQL 기본값 | ✅ 높은 안정성
문제점: select for update에서 Phantom Read 가능
4️⃣ SERIALIZABLE (가장 강함)
모든 SELECT를 SELECT FOR UPDATE로 처리
✅ 완벽한 격리 | ❌ 동시성 최악
문제 없음 (모두 해결) | 성능 심각 악화

2. Isolation Level 문제 분류

① Dirty Read (더티 리드)

❌ 문제: COMMIT되지 않은 데이터를 읽는 것

TX A: 임시 데이터 생성 (아직 COMMIT 안 함)

TX B: 그 임시 데이터 읽음 ← 문제!

TX A: ROLLBACK (임시 데이터 취소)

→ TX B가 읽은 데이터가 사라짐!

해결: READ COMMITTED 이상

② Non-Repeatable Read (반복 불가능한 읽기)

❌ 문제: 같은 조건으로 조회할 때마다 다른 데이터

TX B: 데이터 조회 (예: 고액 자산가 0명)

TX A: 데이터 변경 후 COMMIT (고액 자산가 추가)

TX B: 같은 조건 재조회 (예: 고액 자산가 1명) ← 다른 결과!

→ 같은 TX 내에서 읽기 불일치!

해결: REPEATABLE READ 이상

③ Phantom Read (팬텀 리드)

❌ 문제: 새로운 행이 갑자기 나타남

TX B: SELECT FOR UPDATE로 조회 (중위권 자산가 1명)

TX A: 새로운 행 INSERT (고액 자산가 300원 추가) + COMMIT

TX B: SELECT FOR UPDATE 재조회 (어? 2명이 나옴!) ← 새 행 감지!

→ 없던 행이 갑자기 조회됨!

해결: SERIALIZABLE (또는 MySQL InnoDB의 next key lock)

3. READ UNCOMMITTED 테스트

⚠️ 설정 확인:
TX B: SET @@session.transaction_isolation='READ-UNCOMMITTED';

시나리오: 계좌 조회 vs 계좌 업데이트

Transaction A (편집) Transaction B (조회)
SELECT * FROM account;
(no=1: 100원)		 SELECT * FROM account WHERE amount>=200;
→ 고액 자산가 0명
UPDATE account SET amount=200 WHERE no=1;
SELECT * FROM account;
(no=1: 200원 - 임시상태)		 TX A가 UPDATE 시작 (COMMIT 전)
아직 COMMIT 안 함 SELECT * FROM account WHERE amount>=200;
→ 고액 자산가 1명! 🚨
(TX A의 임시 데이터 읽음)
💥 Dirty Read 발생!
TX B가 이 데이터를 기반으로 다른 작업을 했는데, TX A가 ROLLBACK하면?
→ TX B가 읽은 데이터가 존재하지 않음 (데이터 무결성 위반)

4. READ COMMITTED 테스트

⚠️ 설정 확인:
TX B: SET @@session.transaction_isolation='READ-COMMITTED';

시나리오: 같은 TX 내에서 다른 데이터 조회

Transaction A (편집) Transaction B (조회)
SELECT * FROM account;
(no=1: 100원)		 SELECT * FROM account WHERE amount>=200;
→ 고액 자산가 0명
UPDATE account SET amount=200 WHERE no=1;
(아직 COMMIT 전)		 —
COMMIT;
(변경사항 확정)		 —
— SELECT * FROM account WHERE amount>=200;
→ 고액 자산가 1명! 🚨
(TX A의 COMMIT 후 새 데이터)
문제 상황:
TX B는 아무것도 안 하고 조회만 했는데, 같은 조건의 조회가 다른 결과를 반환함!
긴 업무를 진행 중인 TX B에서 조회 불일치 발생
💥 Non-Repeatable Read 발생!
같은 TX 내에서 같은 쿼리를 실행했는데 다른 결과
✅ Dirty Read는 해결됨:
COMMIT 전의 데이터는 읽지 않음 (임시 데이터 차단)

5. REPEATABLE READ 테스트

⚠️ 설정 확인:
TX B: SET @@session.transaction_isolation='REPEATABLE-READ';
MySQL 기본값이므로 생략 가능

동작 원리: UNDO 영역 활용

핵심: 일반적인 SELECT는 UNDO 영역에서 데이터를 조회

  • 현재 TX 시작 후 변경된 데이터 → UNDO 영역에서 변경 전 데이터 조회
  • TX-ID를 통해 어느 시점의 데이터인지 판단
  • 결과적으로 읽은 데이터가 일관성 유지

시나리오: TX-ID를 통한 일관성 보장

Transaction A (TX-ID: 3) | 편집 Transaction B (TX-ID: 2) | 조회
SELECT * FROM account; SELECT * FROM account WHERE amount>=200;
→ 고액 자산가 0명
UPDATE account SET amount=200 WHERE no=1;
(테이블에 임시 저장, UNDO에 백업)		 TX A가 UPDATE 시작
테이블의 값은 TX-ID가 낮으므로
스킵, UNDO에서 기존 자료 조회
(아직 COMMIT 전)		 SELECT * FROM account WHERE amount>=200;
→ 여전히 0명! ✅
(TX B 기준: UNDO 영역에서 조회)
COMMIT; —
— SELECT * FROM account WHERE amount>=200;
→ 여전히 0명! ✅
(최초 읽었던 데이터 유지)
— COMMIT;
(TX B 종료, TX-ID: 4로 변경)
— SELECT * FROM account WHERE amount>=200;
→ 1명! 📊
(새로운 TX에서는 테이블 직접 조회)
✅ Non-Repeatable Read 해결!
같은 TX 내에서 같은 조건 조회 = 같은 결과 (UNDO 영역 활용)

⚠️ Phantom Read (SELECT FOR UPDATE)

Transaction B (TX-ID: 2) Transaction A (TX-ID: 3)
SELECT * FROM account WHERE amount>=100;
→ 중위권 자산가 1명		 —
SELECT * FROM account WHERE amount>=100 FOR UPDATE;
(DML 준비 - 실제 테이블 조회)		 —
— INSERT INTO account (name, amount) VALUES('jang gil san', 300);
COMMIT;
SELECT * FROM account WHERE amount>=100 FOR UPDATE;
→ 2명! 🚨
(새로운 행 감지 - Phantom Read)		 —
💥 Phantom Read 발생!
이유: SELECT FOR UPDATE는 실제 테이블에서 조회 (UNDO 영역 미사용)
→ DML을 위해 Lock을 잡아야 하므로 UNDO 사용 불가

MySQL InnoDB 해결책: next key lock (row lock + gap lock)
→ 새로운 행 INSERT 자체가 차단됨 (Phantom Read 방지)

6. SERIALIZABLE 테스트

⚠️ 설정 확인:
TX B: SET @@session.transaction_isolation='SERIALIZABLE';

동작 원리: 모든 SELECT가 SELECT FOR UPDATE

핵심: 모든 SELECT가 자동으로 Lock을 획득

  • SELECT: S-Lock 획득 (다른 TX의 DML 차단)
  • INSERT/UPDATE/DELETE: X-Lock 획득 (모든 접근 차단)
  • 결과: 완벽한 격리 (모든 문제 해결)
  • 단점: 동시성 최악 (거의 순차 처리)

시나리오: 완벽한 격리 = 완벽한 성능 저하

Transaction A Transaction B
— SELECT * FROM account WHERE amount>=200;
→ S-Lock 획득!
INSERT INTO account (name, amount) VALUES('jang gil san', 300);
→ X-Lock 시도...		 TX A의 INSERT 대기 중
ERROR 1205:
Lock wait timeout
(TX B의 S-Lock 때문에 차단!)		 —
— COMMIT;
(S-Lock 반납)
INSERT INTO account (name, amount) VALUES('jang gil san', 300);
→ 이제 성공!		 —
✅ 모든 문제 해결!
Dirty Read ✓ | Non-Repeatable Read ✓ | Phantom Read ✓
💥 성능 문제!
단순 조회도 Lock을 잡음 → 다른 작업 모두 대기
거의 순차 처리 수준으로 동시성 완전 상실

8. 선택 가이드

✅ 일반적 선택: READ COMMITTED 또는 REPEATABLE READ
READ COMMITTED: Oracle 사용 환경, 높은 동시성 필요
REPEATABLE READ: MySQL 환경, 중간 수준의 안정성과 동시성 필요
❌ 피해야 할 선택
READ UNCOMMITTED: 거의 모든 경우 사용 금지 (너무 위험)
SERIALIZABLE: 성능이 매우 중요한 경우 피함 (거의 사용 안 함)
