BertWordPieceTokenizer

학습되지 않은 빈 토크나이저를 생성하고, WordPiece 알고리즘으로 서브워드를 분리하는 과정을 시각화합니다.

🗂️ Tokenizer 종류 및 역할
BertWordPieceTokenizer 현재
사용 모델: BERT, DistilBERT, ALBERT
WordPiece 알고리즘으로 단어를 서브워드로 분리합니다. 연속 서브워드에는 ## 접두사를 붙여 구분하며, [CLS] [SEP] 등 특수 토큰을 함께 사용합니다.
📚 훈련(train) 파라미터
vocab_size default: 30,000
WordPiece 학습이 끝났을 때 최종적으로 만들어지는 토큰 목록의 크기입니다. 자주 등장하는 서브워드 쌍을 합치다가 vocab이 이 숫자에 도달하면 학습을 멈춰요.
  • 너무 작으면 단어를 잘게 쪼개야 해서 ##ing ##s 같은 토큰이 많아지고 시퀀스가 길어집니다.
  • 너무 크면 희귀 단어까지 통째로 vocab에 들어가지만 메모리·학습 비용이 증가합니다.
  • BERT 기본값 30,000은 영어 기준 균형점이며, 한국어는 형태소가 다양해 보통 더 큰 값을 사용합니다.
initial_alphabet
학습 전에 미리 vocab에 넣어 둘 문자(alphabet) 목록입니다. 반드시 포함되어야 하는 문자를 사전에 추가할 때 사용해요.
  • initial은 학습 이전에 미리 단어를 vocab에 넣는 것을 의미합니다.
  • special token들도 initial에 포함되어 vocab에 추가됩니다.
limit_alphabet default: 1,000
initial token(문자 단위)의 최대 갯수를 제한합니다. 너무 많은 문자가 초기 vocab에 들어가지 않도록 상한선을 두는 역할이에요.
min_frequency default: 2
서브워드 쌍(pair)을 병합할 때 적용되는 최소 등장 빈도수입니다. 개별 문자(base vocab) 등록에는 적용되지 않아요.
  • 두 서브워드가 이 값 이상 붙어서 등장해야 병합 후 vocab에 추가됩니다.
  • 기본값 2이면 딱 1번만 붙어서 등장한 쌍은 병합되지 않습니다.
  • 개별 문자는 단어를 표현하기 위해 무조건 base vocab에 등록되며, 이 값의 영향을 받지 않습니다.
special_tokens
특수 토큰 목록을 직접 지정할 수 있습니다. BERT는 아래 5개의 스페셜 토큰을 기본으로 사용해요.
  • [PAD] 시퀀스 길이를 맞추기 위한 패딩 토큰
  • [UNK] vocab에 없는 OOV(Out Of Vocabulary) 단어를 대체하는 토큰
  • [CLS] 문장의 시작을 알리며, 분류(Classification) 태스크에서 문장 전체 표현으로 활용
  • [SEP] 두 문장 사이를 구별하는 구분자 토큰
  • [MASK] MLM(Masked Language Modeling) 학습 시 마스킹된 위치를 나타내는 토큰
wordpiece_prefix default: '##'
서브워드임을 표시하는 접두사입니다. BERT는 기본적으로 '##'을 씁니다.
  • 예: SS, ##AF, ##Y → sub-word 구분에 활용
🏗️ Vocab 구축 과정 (학습 단계)
⚠️ 다음 내용은 개념적으로 단순화시킨 것으로, 실제 디테일은 많이 다를 수 있습니다.
WordPiece 학습은 아래 4단계를 거쳐 vocab을 구성합니다. vocab_size에 도달할 때까지 병합을 반복해요.
예시
"고양이가 좋아요"100회
"강아지가 귀여워요"90회
"앵무새를 봤어요"1회
① 스페셜 토큰
먼저 등록
빈도와 무관하게 vocab에 제일 먼저 고정 ID로 추가됩니다.
[PAD] [UNK] [CLS] [SEP] [MASK]
② 개별 문자
전부 등록
학습 데이터에 등장한 모든 개별 문자를 base vocab에 추가합니다. min_frequency와 무관해요.
고 양 이 강 아 지 앵 무 새 …
💡 "앵무새"는 1번만 등장했지만 앵, 무, 새는 모두 base vocab에 등록돼요. WordPiece가 [UNK]를 최소화하는 핵심 원리예요!
③ 문자 쌍
병합
단어를 문자 단위로 쪼갤 때 첫 번째 문자는 그대로, 나머지는 ##을 붙여요. 인접 쌍의 빈도를 계산해 min_frequency 이상인 쌍만 병합합니다.
고##양
180회 ✓ → 고양
강##아
160회 ✓ → 강아
앵##무
1회 ✗ → 병합 안 됨
④ 더 긴
서브워드 병합
병합된 서브워드끼리 다시 빈도를 계산해 더 긴 서브워드를 만들어요. vocab_size에 도달하면 종료.
1 고양+##이 → 고양이 180회 ✓
2 강아+##지 → 강아지 160회 ✓
3 고양이+##가 → 고양이가 100회 ✓
min_frequency는 문자 등록이 아닌 쌍 병합 조건으로만 작용해요. 개별 문자는 학습 데이터에 등장하는 순간 무조건 base vocab에 들어갑니다.
🔖 스페셜 토큰 사용
[CLS] / [SEP] — 문장 구조
[CLS] 강아지가 귀여워요 [SEP]
🎯 스페셜 토큰이 문장의 시작과 끝 구조를 잡아줘요!
[UNK] — 평가 시 활용 · "我爱你" (학습 데이터에 없는 문자)
[UNK] [UNK] [UNK]
❌ base vocab에 없는 문자는 모두 [UNK]로 처리됩니다. 학습 때는 거의 등장하지 않지만, 평가·추론 시 미등록 문자를 만날 때 활용됩니다.
[PAD] — 배치 처리 시 길이 맞추기
문장 1: "고양이가 좋아요" (2토큰)
고양이가 좋아요
문장 2: "토끼" (1토큰) → 최대 길이 2로 패딩
토끼 [PAD]
📦 배치로 여러 문장을 묶을 때 길이를 맞추기 위해 짧은 문장 뒤에 [PAD]를 채워요.
🔢 토큰 ID 변환
토크나이저는 각 토큰을 vocab에서의 인덱스(정수)로 바꿔요. 원핫 인코딩이 아니에요!
텍스트 → 토큰 → ID
"고양이가 좋아요" →
고양이가 좋아요
→ [1523, 892]
"강아지가 귀여워요" →
강아지가 귀여워요
→ [884, 731]
💡 vocab에서 해당 토큰의 위치(행 번호)를 그대로 꺼내는 거예요. 복잡한 계산 없이 단순 룩업이에요!
⚖️ 원핫 인코딩 vs nn.Embedding
토큰 ID를 벡터로 바꾸는 방법은 두 가지가 있어요. 왜 원핫을 안 쓰는지 비교해봐요.
❌ 원핫 인코딩
고양이가 (ID=1523)
→ [0, 0, ..., 1, ..., 0]
(30,000차원!)
• vocab 크기만큼 차원 폭발 • 대부분이 0 → 비효율적 • 단어 간 의미 관계 표현 불가
✅ nn.Embedding
고양이가 (ID=1523)
→ [0.12, -0.34, 0.88, ...]
(768차원, 밀집 벡터)
• 고정된 작은 차원 (보통 768) • 모든 값이 의미 있음 • 학습을 통해 의미 관계 반영
📖 nn.Embedding 동작 원리
내부적으론 원핫 × 가중치행렬과 수학적으로 동일하지만, 실제론 인덱스로 행을 직접 꺼내는 룩업 테이블이에요.
코드 예시
import torch
import torch.nn as nn

embedding = nn.Embedding(30000, 768)
token_ids = torch.tensor([5, 1523, 892, 6])
# [CLS] 고양이가 좋아요 [SEP]
vectors = embedding(token_ids)
# → [4, 768] 텐서 [토큰 수, 임베딩 차원]

🤖 BERT 학습 — MLM (Masked Language Modeling)
BERT는 Tokenizer가 만든 vocab으로 변환된 문장에서 일부 토큰을 [MASK]로 가리고 맞추는 방식으로 언어를 학습해요.
① 원본 문장 → 토크나이저 변환
"고양이가 생선을 먹어요"
고양이가 생 선 을 먹 어 요
Token IDs: [1523, 8821, 4532, 221, 9012, 334, 219]
② 랜덤 15% 마스킹 → "생" 토큰을 [MASK]로 대체
고양이가 [MASK] 선 을 먹 어 요
입력 IDs: [1523, 4, 4532, 221, 9012, 334, 219]
정답 label: [-100, 8821, -100, -100, -100, -100, -100]
💡 -100은 "손실 계산 안 함", 8821은 "생"의 ID
📈 BERT 예측 & 가중치 업데이트
BERT가 문제 풀기
📥 입력: [고양이가] [MASK] [선] [을] [먹] [어] [요]
🧠 "고양이가" 다음 + "선"과 함께 "○선" + "먹어요"와 연관 → vocab 30,000개 각각 확률 계산
ID 1: "고" → 0.001% ID 2: "양" → 0.003% ... ID 8821: "생" → 85.2% ✅ ID 8822: "선" → 2.1%
✅ 맞혔을 때
예측 8821("생") = 정답 8821("생") → Loss 낮음
100% 확률이 아니므로 약간의 가중치 조정 → 다음엔 더 확신 있게 예측
❌ 틀렸을 때
예측 4532("선") ≠ 정답 8821("생") → Loss 높음
역전파로 가중치 크게 업데이트 → "고양이가 다음엔 '선'보다 '생'이 올 확률이 높다" 학습!
