🔄 인코딩

정의: 데이터를 다른 포맷으로 변환

특징:

→ 원본 복원 가능
→ 규칙/알고리즘 공개
→ 보안 목적 X

예: Base64, URL Encoding, UTF-8

#️⃣ 해싱

정의: 데이터를 고정길이 해시값으로 변환

특징:

→ 단방향 함수
→ 작은 변화도 완전히 다른 해시
→ 복원 불가능

예: MD5, SHA-256

🔐 암호화

정의: 키를 이용해 데이터를 기밀화

특징:

→ 양방향 (암호화/복호화)
→ 키가 있어야 복원 가능
→ 보안 성능 최우선

예: AES, RSA

✍️ 서명

정의: 개인키로 데이터의 진위성 증명

특징:

→ 발신자 인증
→ 서명만으로는 데이터 보호 X
→ 부인 불가 보장

예: RSA 서명, ECDSA

💼 실제 사용처

인코딩 (Encoding)
→ 데이터 전송/저장 형식 변환
예: URL 인코딩, Base64 (이미지 전송), HTML 엔티티, UTF-8 문자 인코딩
해싱 (Hashing)
→ 비밀번호 저장 / 무결성 검증 / 중복 검사
예: 비밀번호 저장 (bcrypt/scrypt), 파일 무결성 검증, 캐시 키, 블록체인, Git 커밋 ID
암호화 (Encryption)
→ 데이터 기밀성 보호 / 통신 보안
예: HTTPS/TLS (AES), API 키 저장, 결제 정보 암호화, 메시지 앱 통신, 데이터베이스 암호화
서명 (Signing)
→ 발신자 인증 / 부인 방지 / 무결성 확인
예: 디지털 서명, JWT 토큰, SSL 인증서, 문서 전자 서명, API 요청 검증
핵심 정리:
• 인코딩 = 형식 변환 (복원 O, 보안 X)
• 해싱 = 무결성 검증 (복원 X, 단방향)
• 암호화 = 기밀성 보호 (복원 O, 키 필요)
• 서명 = 진위성 증명 (복원 X, 발신자 확인)

🎯 상황별 선택 가이드

전송 형식만 변환하고 싶을 때?
→ 인코딩 (Base64, URL Encoding)
데이터가 변조되지 않았는지 확인하고 싶을 때?
→ 해싱 (파일 해시, 비밀번호)
데이터를 안전하게 숨기고 나중에 다시 쓸 때?
→ 암호화 (AES, RSA로 암호화 후 복호화)
이 메시지가 정말 그 사람이 보낸 것인지 증명하고 싶을 때?
→ 서명 (디지털 서명, JWT)
