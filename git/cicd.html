# 권장 디렉토리 구성
.github/
├── workflows/                  # ← Workflow 파일 위치 (필수)
│   ├── ci.yml                  # CI: 빌드 + 테스트

│       └── action.yml
└── CODEOWNERS                  # 코드 오너 설정

# 확장자 → .yml 또는 .yaml 모두 가능

SECTION 03
Workflow 기본 구조
핵심 구성 요소
  • nameWorkflow 이름 → GitHub Actions UI에 표시
  • on트리거 조건 → push, pull_request, schedule 등
  • env전역 환경변수 → 모든 job에서 사용 가능
  • jobs실행 단위 → 병렬 또는 순서 실행 가능
  • runs-onRunner 환경 → ubuntu-latest, windows-latest, macos-latest
  • steps실제 실행 명령 → uses(액션) 또는 run(쉘 명령)
.github/workflows/ci.yml
name: CI Pipeline

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

env:
  NODE_VERSION: '20'

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - name: 코드 체크아웃
        uses: actions/checkout@v4
      - name: Node.js 설정
        uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'npm'
      - run: npm ci
      - run: npm run lint && npm test && npm run build

SECTION 04
트리거(on) 설정
push
브랜치 push 시 → branches, tags, paths 필터 지원
pull_request
PR 생성·수정·동기화 시 → types로 이벤트 세분화
schedule
cron 표현식으로 정기 실행 → UTC 기준
workflow_dispatch
수동 실행 버튼 → UI/API에서 직접 트리거
release
GitHub Release 생성 시 → published, created 등
workflow_call
다른 Workflow에서 호출 → 재사용 가능한 Workflow
트리거 예시 (고급)
on:
  push:
    branches: [ main, 'release/**' ]
    paths: [ 'src/**', 'package*.json' ]
    paths-ignore: [ 'docs/**' ]
  pull_request:
    types: [ opened, synchronize, reopened ]
  schedule:
    - cron: '0 2 * * 1-5'   # 평일 오전 2시 (UTC)
  workflow_dispatch:
    inputs:
      environment:
        description: '배포 환경'
        required: true
        type: choice
        options: [ staging, production ]

SECTION 05
Jobs & Steps 심화
컨텍스트란?
  • ${{ 컨텍스트.속성 }} 형식 → Workflow 실행 중 각종 값에 접근하는 표현식
  • 컨텍스트 종류에 따라 GitHub 정보·시크릿·이전 스텝 결과 등 참조 가능
주요 컨텍스트 참조 예시
env:
  NODE_VERSION: '20'

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: github 컨텍스트 — GitHub 실행 정보
        run: |
          echo "브랜치: ${{ github.ref_name }}"        # 현재 브랜치명
          echo "커밋: ${{ github.sha }}"             # 커밋 SHA
          echo "이벤트: ${{ github.event_name }}"    # push / pull_request 등
          echo "저장소: ${{ github.repository }}"    # owner/repo 형식

      - name: env 컨텍스트 — 환경변수 참조
        uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}     # 위 env 블록에서 정의한 값

      - name: secrets 컨텍스트 — 시크릿 참조
        env:
          API_KEY: ${{ secrets.API_KEY }}           # Settings에서 등록한 시크릿
          TOKEN:   ${{ secrets.GITHUB_TOKEN }}      # GitHub 자동 발급 토큰
        run: curl -H "Authorization: Bearer $TOKEN" https://api.example.com

      - name: 출력값 생성
        id: meta                                       # ← id 지정 필수
        run: echo "image-tag=v1.0.${{ github.run_number }}" >> $GITHUB_OUTPUT

      - name: steps 컨텍스트 — 이전 스텝 출력값 참조
        run: echo "태그: ${{ steps.meta.outputs.image-tag }}"  # steps.{id}.outputs.{key}
컨텍스트 종류 요약
컨텍스트주요 속성참조 예시
github브랜치·커밋·이벤트·저장소 정보${{ github.ref_name }}
envenv: 블록에서 정의한 변수${{ env.NODE_VERSION }}
secretsSettings에서 등록한 시크릿${{ secrets.API_KEY }}
steps이전 스텝의 outputs${{ steps.myid.outputs.val }}
needs이전 Job의 outputs${{ needs.build.outputs.tag }}
inputsworkflow_dispatch 입력값${{ inputs.environment }}
needs — Job 순서 제어
jobs:
  test:                      # 1단계
    runs-on: ubuntu-latest
  build:
    needs: test              # 2단계: test 완료 후
    outputs:
      image-tag: ${{ steps.meta.outputs.tags }}
  deploy:
    needs: [ test, build ]   # 3단계: 복수 의존성
    steps:
      - run: echo "${{ needs.build.outputs.image-tag }}"
if 조건 — 브랜치/이벤트별 분기
jobs:
  deploy-staging:
    if: github.ref == 'refs/heads/develop'
  deploy-production:
    if: github.ref == 'refs/heads/main'
    steps:
      - if: always()          # 이전 실패해도 항상 실행
        run: echo "클린업"
      - if: failure()         # 실패 시에만
        run: echo "Slack 알림"
Steps 주요 키워드
키워드설명예시
usesMarketplace 액션 사용 → github.com/marketplaceactions/checkout@v4
run쉘 명령 직접 실행npm test
with액션 입력값 전달node-version: '20'
env스텝 전용 환경변수API_KEY: ${{ secrets.KEY }}
id스텝 ID → 출력값 참조 시steps.myid.outputs.val
continue-on-error오류 무시 후 진행true / false
timeout-minutes스텝 최대 실행 시간10

SECTION 06
환경변수 & 시크릿 관리
⚠️
시크릿은 코드에 직접 입력 금지 → Settings → Secrets and variables → Actions에서 등록
시크릿 등록 위치 보안
  • Repository Secrets: 단일 저장소 전용
  • Environment Secrets: 환경(prod/stage)별 분리
  • Organization Secrets: 조직 전체 공유
  • 참조: ${{ secrets.SECRET_NAME }}
환경변수 우선순위
  • Step env → 가장 높은 우선순위
  • Job env → Step 하위에 적용
  • Workflow env → 전역 기본값
  • 참조: ${{ env.VAR_NAME }}
OIDC 인증 — AWS 액세스 키 없이 배포
permissions:
  id-token: write           # OIDC 토큰 발급
  contents: read

jobs:
  deploy:
    environment: production
    steps:
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789:role/GitHubActionsRole
          aws-region:     ap-northeast-2
          # 장기 자격증명 없이 임시 토큰 자동 발급

SECTION 07
실전 예제 모음
.github/workflows/docker-deploy.yml
name: Docker Build & Deploy
on:
  push:
    branches: [ main ]
jobs:
  build-and-push:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: ghcr.io/${{ github.repository }}:${{ github.sha }}
          cache-from: type=gha
          cache-to:   type=gha,mode=max
.github/workflows/node-aws.yml
name: Node.js CI/CD → AWS ECS
on:
  push:
    branches: [ main ]
jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20', cache: 'npm' }
      - run: npm ci && npm run lint && npm test
  deploy:
    needs: ci
    environment: production
    runs-on: ubuntu-latest
    steps:
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}
          aws-region:     ap-northeast-2
      - run: aws ecs update-service --cluster prod --force-new-deployment
.github/workflows/release.yml
name: 자동 릴리즈
on:
  push:
    tags: [ 'v*.*.*' ]       # v1.0.0 형식 태그 시 실행
jobs:
  release:
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0        # 전체 히스토리 필요
      - run: npm ci && npm run build
      - uses: softprops/action-gh-release@v2
        with:
          generate_release_notes: true
          files: dist/**

보안 체크리스트
필수 보안 설정
  • ✓permissions 명시적 선언 → 기본값 write 방지
  • ✓외부 액션 SHA 핀닝 → @v4 대신 @abc1234
  • ✓시크릿 로그 출력 금지 → echo $SECRET 절대 금지
  • !Dependabot 활성화 → 액션 자동 버전 업데이트
  • iOIDC 활용 → AWS/GCP 자격증명 토큰 기반 인증
Branch Protection 설정
  • ✓main 브랜치 직접 push 차단
  • ✓PR 머지 전 CI 통과 필수 설정
  • ✓Required reviewers 최소 1명 지정
  • !CODEOWNERS 파일로 파일별 소유자 지정
  • iEnvironment 보호 규칙 → prod 배포 수동 승인

.gitlab/

SECTION 03
Pipeline 기본 구조
핵심 구성 요소
  • stages파이프라인 단계 순서 정의 → 나열 순서대로 실행
  • image실행 Docker 이미지 → 전역 또는 Job별 지정 가능
  • variables전역 환경변수 → 모든 Job에서 사용
  • before_script각 Job 실행 전 공통 명령 → 의존성 설치 등
  • script실제 실행 명령 목록 → Job의 핵심 (필수)
  • after_scriptJob 완료 후 실행 → 성공·실패 관계없이 실행
  • artifactsJob 결과물 저장 → 이후 Stage에서 다운로드 가능
  • cache의존성 캐싱 → paths + key로 캐시 전략 설정
.gitlab-ci.yml — 기본 구조
stages:               # 실행 순서 정의
  - build
  - test
  - deploy

image: node:20-alpine   # 전역 Docker 이미지

variables:             # 전역 변수
  NODE_ENV: production
  FF_USE_FASTZIP: "true"  # 빠른 아티팩트 압축

cache:                 # 전역 캐시 설정
  key: ${CI_COMMIT_REF_SLUG}
  paths:
    - node_modules/
  policy: pull-push   # pull: 읽기만 / push: 쓰기만 / pull-push: 둘다

build-job:
  stage: build
  before_script:
    - npm ci
  script:
    - npm run build
  artifacts:           # 빌드 결과물 저장
    paths:
      - dist/
    expire_in: 1 week  # 아티팩트 보관 기간

test-job:
  stage: test
  script:
    - npm run lint
    - npm test -- --coverage
  coverage: '/Statements\s*:\s*(\d+\.?\d*)%/'  # 커버리지 파싱 정규식

deploy-job:
  stage: deploy
  environment:
    name: production
    url: https://app.example.com
  script:
    - echo "Deploying to production"

SECTION 04
트리거 설정
⚠️
only / except는 레거시 → 신규 프로젝트는 rules 사용 권장
rules
조건부 실행 (권장) → if, changes, exists 조합
only / except
브랜치·태그 필터 (레거시) → rules로 마이그레이션 권장
when
실행 조건 → on_success, on_failure, always, manual, delayed
pipeline triggers
API / 다른 프로젝트에서 파이프라인 직접 트리거
schedules
GitLab UI에서 cron 스케줄 설정 → CI/CD → Schedules
needs
Stage 경계를 넘는 Job 의존성 → DAG 파이프라인 구성
rules 활용 예시 (권장 방식)
deploy-staging:
  stage: deploy
  script:
    - ./deploy.sh staging
  rules:
    - if: $CI_COMMIT_BRANCH == "develop"   # develop 브랜치 push 시

deploy-production:

GitLab 내장변수란?

  stage: build
  script:
    # $CI_COMMIT_xxx — 커밋 정보

    - curl -H "JOB-TOKEN: $CI_JOB_TOKEN" "$CI_API_V4_URL/projects/$CI_PROJECT_ID/packages"

SECTION 06
변수 & 시크릿 관리
⚠️
민감 변수는 Settings → CI/CD → Variables에서 등록 → Masked 옵션 필수 체크
변수 등록 위치 보안
  • Project Variables: 프로젝트 전용 → 가장 많이 사용
  • Group Variables: 그룹 내 전체 프로젝트 공유
  • Instance Variables: GitLab 인스턴스 전체 (관리자 전용)
  • 참조: $VARIABLE_NAME 또는 ${VARIABLE_NAME}
변수 보호 옵션
  • Masked: 로그에서 값 숨김 → 민감 정보 필수
  • Protected: 보호된 브랜치·태그에서만 사용
  • Expanded: 다른 변수 참조 가능 여부
  • File type: 파일로 전달 → 인증서·키 파일 등
내장 CI/CD 변수 주요 목록
# GitLab이 자동으로 제공하는 내장 변수
$CI_COMMIT_SHA          # 현재 커밋 SHA (40자)
$CI_COMMIT_SHORT_SHA    # 커밋 SHA 단축 (8자)
$CI_COMMIT_BRANCH       # 현재 브랜치 이름
$CI_COMMIT_TAG          # 현재 태그 이름 (태그 파이프라인 시)
$CI_COMMIT_REF_SLUG     # URL 안전 브랜치명 (슬러그)
$CI_PIPELINE_SOURCE     # 파이프라인 트리거 원인 (push, web, schedule...)
$CI_PROJECT_NAME        # 프로젝트 이름
$CI_REGISTRY_IMAGE      # GitLab Container Registry 주소
$CI_REGISTRY_USER       # Container Registry 로그인 사용자
$CI_REGISTRY_PASSWORD   # Container Registry 로그인 패스워드
$CI_JOB_TOKEN           # Job 실행 중 임시 토큰 → API 호출 시 활용
$GITLAB_USER_NAME       # 파이프라인 트리거한 사용자 이름

SECTION 08
배포 전략
🔵

Environment + when: manual

  • 프로덕션 Job에 when: manual 설정
  • GitLab UI에서 버튼 클릭으로 승인
  • 승인자 제한 → Protected Environments
  • 배포 이력 자동 기록
🌊

Review Apps (동적 환경)

  • MR마다 독립적인 환경 자동 생성
  • 브랜치명 기반 URL 자동 부여
  • MR 닫힘 시 환경 자동 삭제
  • environment: name: review/$CI_COMMIT_REF_SLUG
🔄

Auto DevOps

  • .gitlab-ci.yml 없이 자동 파이프라인
  • 빌드·테스트·보안·배포 자동 구성
  • Kubernetes 클러스터 연동 시 활성화
  • 커스터마이즈 가능 → 점진적 전환
Review Apps 구성 예시
deploy-review:
  stage: deploy
  script:
    - ./deploy.sh review $CI_ENVIRONMENT_SLUG
  environment:
    name: review/$CI_COMMIT_REF_SLUG    # 브랜치별 독립 환경
    url: https://$CI_ENVIRONMENT_SLUG.example.com
    on_stop: stop-review
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"

stop-review:                          # MR 닫힐 때 환경 정리
  stage: deploy
  script:
    - ./teardown.sh $CI_ENVIRONMENT_SLUG
  environment:
    name: review/$CI_COMMIT_REF_SLUG
    action: stop
  when: manual

SECTION 09
보안 체크리스트
✅
GitLab Ultimate 플랜에서 SAST·DAST·Dependency Scanning 내장 제공 → include: template: Security/SAST.gitlab-ci.yml으로 즉시 활성화
필수 보안 설정
  • ✓민감 변수 Masked + Protected 옵션 활성화
  • ✓Protected Branches → main 직접 push 차단
  • ✓Protected Environments → 프로덕션 배포 승인자 지정
  • !$CI_JOB_TOKEN 권한 범위 최소화 설정
  • iRunner 태그로 민감 Job은 전용 Runner 격리
보안 스캔 자동화
  • ✓SAST → 소스코드 정적 분석
  • ✓Dependency Scanning → 취약 패키지 탐지
  • !Container Scanning → Docker 이미지 취약점
  • !Secret Detection → 코드 내 시크릿 노출 탐지
  • iDAST → 배포된 앱 동적 보안 테스트
보안 스캔 템플릿 포함 예시
include:

  - template: Security/Container-Scanning.gitlab-ci.yml

stages:
  - build
  - test
  - .pre              # 템플릿이 사용하는 내장 stage
  - deploy

variables:

  DS_EXCLUDED_ANALYZERS: "bundler-audit"
